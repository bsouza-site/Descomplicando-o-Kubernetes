<h2 align="center"> Descomplicando os StatefulSets e os Services no Kubernetes. </h2>

<h3 align="center"> 07.1.	O que é um StatefulSet? </h3>

Os StatefulSets são uma funcionalidade do Kubernetes que gerencia o deployment e o scaling de um conjunto de Pods, fornecendo garantias sobre a ordem de deployment e a singularidade desses Pods.

Diferente dos Deployments e Replicasets que são considerados stateless (sem estado), os StatefulSets são utilizados quando você precisa de mais garantias sobre o deployment e scaling. Eles garantem que os nomes e endereços dos Pods sejam consistentes e estáveis ao longo do tempo.

**O que é um StatefulSet no Kubernetes?**  
StatefulSet é um tipo de controlador usado para rodar aplicações stateful (com estado persistente), onde:

* Cada pod tem identidade fixa;
* Cada pod tem seu próprio volume persistente;
* A ordem de criação e remoção importa;
* Nomes e DNS são previsíveis.

Diferente de Deployments, onde os Pods são descartáveis e idênticos.

<h3 align="center"> 07.5. O que são os Services? </h3>

Os Services no Kubernetes são uma abstração que define um conjunto lógico de Pods e uma política para acessá-los. Eles permitem que você exponha uma ou mais Pods para serem acessados por outros Pods, independentemente de onde eles estejam em execução no cluster.

Os Services são definidos usando a API do Kubernetes, normalmente através de um arquivo de manifesto YAML.

Existem quatro tipos principais de Services:
* **ClusterIP**: é o padrão. Expõe o Service em um IP interno no cluster. Este tipo torna o Service acessível apenas dentro do Cluster.  
* **NodePort**: expõe o Service na mesma porta de cada Node selecionado no cluster usando NAT. Torna o Service acessível de fora do Cluster.  
* **LoadBalancer**: Cria um balanceador de carga externo no ambiente de nuvem atual (se suportado) e atribui um IP fixo, externo ao cluster, ao Service.  
* **ExternalName**: mapeia o Service para o conteúdo do campo externalName (por exemplo, foo.bar.example.com), retornando um registro CNAME com seu valor.  

**Como os Services funcionam?**  
Os Services no Kubernetes fornecem uma abstração que define um conjunto lógico de Pods e uma política para acessá-los. Os conjuntos de Pods são determinados por meio de seletores de rótulo (Label Selectors). Embora cada Pod tenha um endereço IP único, esses Ips não são expostos fora do cluster sem um serviço.

Sempre é bom reforçar a importância dos Labels no Kubernetes, pois eles são a base para a maioria das operações no Kubernetes, então cuide com carinho dos seus Labels.  

<h3 align="center"> 07.6. Criando os nossos Services ClusterIP e NodePort. </h3>

Para criar um exemplo básico para expor um serviço com ClusterIP, execute os passos abaixo:  
* **Crie um deployment**: kubectl create deployment nginx --image nginx --port 80  
* **Expor o serviço**: kubectl expose deployment nginx --type NodePort  
* **Para ver mais detalhes desse serviço**: kubectl describe svc nginx  
<p align="center"><img width="468" height="289" alt="image" src="https://github.com/user-attachments/assets/a6cfaf8d-859e-400f-a226-b4b3e02da806" /></p>

**kubectl expose**  
O comando **kubectl expose** é um atalho do Kubernetes para criar um **Service** rapidamente a partir de um recurso que já existe (pod, service, replicationcontroller, deployment ou replicaset).  

**O que é Endpoints em um Service no Kubernetes?**  
Endpoints são a lista real de IPs e portas dos Pods que um Service vai encaminhar o tráfego.  

**Como isso funciona na prática?**  
* Você cria um Pod / Deployment.  
* Os Pods recebem os labels.  
* Você cria um Service com **selector**.  
* O Kubernetes cria automaticamente um objeto **Endpoints**.  
* Esse objeto contém os IPs dos Pods que casam com o **selector**.  

**Para ver os endpoints existentes, execute:** kubectl get endpoints

**ExternalName.**  
Um Service **ExternalName** no Kubernetes é um service que não aponta para Pods. Ele funciona como um alias DNS dentro do Kubernetes, redirecionando o nome do Service para um hostname externo.
