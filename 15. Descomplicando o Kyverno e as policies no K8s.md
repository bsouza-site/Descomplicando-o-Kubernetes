<h2 align="center"> Descomplicando o Kyverno e as policies no K8s. </h2>

<h3 align="center"> 15.1. O que é o Kyverno? </h3>

O Kyverno é uma ferramenta de policy engine (motor de políticas) nativa para o Kubernetes. Ele serve para definir, validar e automatizar regras dentro do cluster, sem precisar escrever código complexo.

Em vez de usar linguagens como Rego (do Open Policy Agent), o Kyverno usa YAML, que é o mesmo formato já utilizado no Kubernetes - isso facilita muito a adoção.

As principais funções do Kyverno são:

**1. Validação de Recursos**: Verifica se os recursos do Kubernetes estão em conformidade com as políticas definidas. Por exemplo, pode garantir que todos os Pods tenham limites de CPU e memória definidos.

**2. Mutação de Recursos**: Modifica automaticamente os recursos do Kubernetes para atender às políticas definidas. Por exemplo, pode adicionar automaticamente labels específicos a todos os novos Pods.

**3.	Geração de Recursos**: Cria recursos adicionais do Kubernetes com base nas políticas definidas. Por exemplo, pode gerar NetworkPolicies para cada novo Namespace criado.  

Veja mais informações na página do **[Kyverno](https://kyverno.io/)**.

Nesta aula, será criado uma Policy para que, apenas pods com limites de recursos definidos sejam implantados no cluster Kubernetes.

Para iniciar a criação da Policy, inicie o arquivo **15_require-resource-limits.yaml** conforme exemplo abaixo:
<p align="center"><img width="446" height="354" alt="image" src="https://github.com/user-attachments/assets/a501f6b2-09f9-44f7-91f2-5764dc0b322f" /></p>

* **Aplique essa configuração**: kubectl apply -f 15_require-resource-limits.yaml  

Abaixo, a explicação de cada campo:  
* **apiVersion**: versão da API do Kyverno.  
* **kind: ClusterPolicy** é uma política para o cluster todo. Para usar dentro de um namespace especifico, use **Policy**.  
* **metadata**: informações. O campo **name** indica o nome da policy.  
* **spec**: especificações da policy.  
  * **validationFailureAction**: usando **Enforce**, bloqueia a criação do recurso se não atender a regra. Também pode ser usado **Audit**, que apenas alerta, não bloqueia a criação.
  * **rules**: nome da regra dentro da policy.  
  * **match**: em **match**, **resources**, **kind**, é o local onde a regra se aplica, nesse caso, **Pods**.  
* **validate**: em **message**, informe uma mensagem que é exibida quando a validação falhar.  
* **pattern**: tudo dentro de **pattern** indica o padrão de Pods exigidos. Os campos com asterisco indicam:  
  * **"*"** qualquer nome de container.  
  * **"?*"** campo obrigatório e não vazio.  

🚨 essa policy ajuda a:  

* Evitar consumo descontrolado de recursos;  
* Melhorar estabilidade do cluster;  
* Garantir boas práticas de deploy;  
* Evitar pods "sem limites" (causadores de problemas).

<h3 align="center"> 15.4. Criando uma Policy do tipo Mutate para modificar recursos. </h3>

Uma policy do tipo Mutate no Kyverno é uma regra que modifica automaticamente os recursos do Kubernetes no momento em que eles são criados ou atualizados.  

Diferente da **validate** (que bloqueia) ou **generate** (que cria novos recursos), a **Mutate** altera o próprio recurso enviado.  

Nesta aula, crie o arquivo **15_add-label-ns.yaml** com o conteúdo abaixo:  
<p align="center"> <img width="276" height="260" alt="image" src="https://github.com/user-attachments/assets/5b28c9dc-7664-4f9f-aea7-d874bad8b700" /></p>

* **Aplique essa configuração**: kubectl apply -f 15_add-label-ns.yaml  

Abaixo, a explicação dos novos campos:
* **spec**: especificações da regra. O campo principal é o **kinds**, que indica que a regra será aplicada em um namespace.  
* **mutate**: indica que a policy é do tipo mutate.  
  * **patchStrategicMerge**: define o tipo de patch que será aplicado no recurso.  
  * **metadata** e **labels** indicam a label que será inclusa nos novos namespaces criados.

Uma policy do tipo **Generate** no Kyverno serve para criar automaticamente recursos no Kubernetes com base em outros eventos.

As configurações dessa Policy indicam:  
* **generate**: indica que a regra vai criar automaticamente um novo recurso quando a condição do **match** for atendida (nesse caso, quando um namespace for criado).

  * **kind**: tipo de recurso que será criado.  
  * **name**: nome do recurso.

Para criar uma Policy proibindo a criação de containers com o usuário root, crie o arquivo **15_disallow-root-user.yaml** com o conteúdo abaixo:
<p align="center"><img width="371" height="308" alt="image" src="https://github.com/user-attachments/assets/06576a7f-cff7-4271-8309-8dccd5daeba9" /></p>

* **Aplique essa configuração**: kubectl apply -f 15_disallow-root-user.yaml  

As principais configurações dessa policy indicam:  
* **validate**: define a regra que o Kyverno deve verificar para decidir se permite ou bloqueia a criação de um recurso, nesse caso, um **Pod**.  
* **message**: mensagem de erro caso o recurso seja bloqueado.  
* **pattern**: descreve a estrutura que o recurso deve ter para ser considerado válido.  
  * **spec**: indica que a verificação deve ocorrer dentro da especificação técnica do Pod.  
  * **containers**: a política irá validar todos os containers do Pod.  
  * **securityContext**: este é o campo oficial do Kubernetes onde são definidas as configurações de segurança do container.  
  * **runAsNonRoot**: definido como **true**. Essa é a regra final. Ela exige que esse campo esteja definido explicitamente como **true**.  

Verifique se a Policy foi criada com o comando **kubectl get clusterpolicies.kyverno.io**
