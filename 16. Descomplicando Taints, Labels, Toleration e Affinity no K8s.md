<h2 align="center"> Descomplicando Taints, Labels, Toleration e Affinity no K8s. </h2>

<h3 align="center"> 16.1.  O que são as Taints? </h3>

No ecossistema do Kubernetes, **Taints** (manchas ou marcas) são propriedades aplicadas aos Nodes (nós) que permitem que o nó "repila" um conjunto de Pods.

Enquanto os _Node Selectors_ e _Affinity_ atraem Pods para nós específicos, os Taints funcionam de forma oposta: eles garantem que Pods indesejados não sejam agendados em máquinas que não deveriam recebê-los.

<h3 align="center"> 16.2. Criando Taints nos Nodes. </h3>

Por padrão, o Control Plane é o único que já vem com um Taint configurado. Para ver a Taint, execute **kubectl describe node**
<p align="center"><img width="555" height="18" alt="image" src="https://github.com/user-attachments/assets/6dd1bb01-6cc2-4d43-8456-4a2b53477cff" /></p>

O Taint acima é um dos mecanismos fundamentais de segurança e estabilidade do Kubernetes. Ele é aplicado automaticamente aos **Control Plane Nodes** (antigamente chamados de Master Nodes) para garantir que as cargas de trabalho dos usuários não interfiram nos processos vitais do cluster.  

Os Taints que podem ser aplicados nos nodes são:  

* **NoSchedule**: faz com que o Kubernetes não agende Pods nesse node a menos que eles tenham uma Toleration correspondente.  
* **PreferNoSchedule**: faz com que o Kubernetes tente não agendar, mas não é uma garantia.  
* **NoExecute**: faz com que os Pods existentes sejam removidos se não tiverem uma Toleration correspondente.  

**kubectl taint**  
O comando **kubectl taint** é usado no Kubernetes para adicionar ou remover taints (manchas) em nós (nodes).

Essas taints controlam quais Pods podem ou não ser agendados (rodar) em um determinado nó.

**Executando taint em um node.**  
Para executar o taint em um node, crie um deployment de exemplo:  
* **kubectl create deployment --image nginx --replicas 6 --port 80 nginx-deployment**  

Neste exemplo, 3 pods foram para um node e 3 pods para outro node.  

**Criar o taint em um node.**  
Para criar um taint em um node e colocá-lo em manutenção, execute:  
* kubectl taint node **node_1** manutencao=true:**NoExecute**  

O campo **manutencao=true** é apenas uma mensagem, e pode ser personalizado.  
O campo **NoExecute** no comando acima gera dois efeitos:  

* Impede que novos Pods sejam adicionados a esse node.  
* Pods que já estão no node e não têm toleration serão removidos e transferidos para outro node.  

Para retirar esse Taint desse node, execute o mesmo comando acima, porém, adicione um traço (**-**) ao final do comando **kubectl taint**.  

Ao retirar o Taint **NoExecute** de um node, o Kubernetes não volta os Pods para esse node automaticamente. Para distribuir novamente os Pods nos nodes existentes, execute o **kubectl rollout**.
* kubectl rollout restart deployment nginx-deployment

**NoSchedule.**  
A taint com efeito **NoSchedule** no Kubernetes indica que os pods atuais continuaram existindo, mas nenhum novo Pod será implantado nesse node.  
* kubectl taint node **node_1** manutencao=true:**NoSchedule**  

Após executar esse comando, os novos pods não serão implantados nesse node.

<h3 align="center"> 16.3.  O que são Tolerations. </h3>

As Tolerations no Kubernetes são configurações aplicadas em Pods que permitem que eles ignorem (tolerem) Taints aplicados em Nodes. O Kubernetes usa esse mecanismo para controlar onde os Pods podem rodar.

Para criar Pods com Toleration, coloque um node como **NoSchedule** com uma key **gpu=true**  

* kubectl taint node **node_1 gpu=true:NoSchedule**  

Agora, inicie um novo deployment e inclua a configuração do Tolerations. Essa configuração deve ser inclusa no mesmo nível de **containers**.

As configurações acima indicam:  

* **key**: refere-se à chave do taint aplicada no node, no caso, **gpu**.  
* **operator**: indica que o valor precisa bater exatamente.  
* **value**: o Pod só tolera o taint se o valor for **true**.  
* **effect**: Esse é o tipo de taint que está sendo tolerado.

O trecho criado acima indica que o Pod pode ser agendado em nodes que têm o taint **gpu=true:NoSchedule**

<h3 align="center"> 16.4.  O que são Labels, Affinity e AntiAffinity. </h3>

**Labels**  
**Labels** são pares **chave=valor** usados para identificar e organizar recursos e podem servir para:  
* Selecionar pods;  
* Organizar recursos;  
* Base para regras de agendamento (Affinity).  

**Affinity**    
**Affinity** define preferências ou regras para onde um Pod deve ser agendado. Existem dois tipos principais:  

**1.	Node Affinity**: controla em quais nodes o Pod pode rodar, baseado em labels do node.  
**2.	Pod Affinity**: faz com que um Pod seja agendado próximo de outros Pods.  

**AntiAffinity**  
O **AntiAffinity** é o oposto do Affinity, define onde o Pod NÃO deve rodar.  

<h3 align="center"> 16.5. Adicionando Labels nos Nodes. </h3>

**kubectl label**  
O comando **kubectl label** é usado para adicionar, alterar ou remover labels de recursos no Kubernetes via linha de comando, sem precisar editar o YAML manualmente.  

Para criar labels em nodes, execute os comandos abaixo:

* kubectl label nodes **cluster1-worker node=cluster1-worker1**  
* kubectl label nodes **cluster1-worker2 node=cluster1-worker2**  

Para ver as labels existentes, execute **kubectl get nodes --show-labels**

Para remover uma label de um node no Kubernetes, use o mesmo comando **kubectl label** usado para criar a label, porém, adicionando um traço (**-**) ao final do comando.  

O parâmetro **--show-labels** pode ser usado em diversos comandos do kubectl, como, por exemplo, **kubectl get ns --show-labels**  

<h3 align="center"> 16.6.  Criando Affinity no Pod. </h3>

Usar um Affinity em um Pod no Kubernetes serve para controlar onde esse Pod será agendado, definindo regras ou preferências sobre em quais nodes ele pode rodar ou com quais outros Pods ele deve ficar próximo (ou longe).

Para criar regras de Affinity, inicie o Deployment **16_pod-affinity.yaml** e adicione as regras... essas regras devem ser inclusas no mesmo nível de **containers**.
<p align="center"><img width="437" height="148" alt="image" src="https://github.com/user-attachments/assets/4cf9e361-de13-44b4-9dc2-7d8b3ec8e6a8" /></p>

* **Aplique essa configuração**: kubectl apply -f 16_pod-affinity.yaml  

Abaixo, a explicação de cada parâmetro:

* **affinity**: inicio da regra de Affinity para o Pod.  
* **nodeAffinity**: indica que a regra é baseada em labels dos nodes.  
* **requiredDuringSchedulingIgnoreDuringExecution**: regra obrigatória. Se não cumprir, o Pod não será agendado. Depois que o Pod estiver rodando, se a label mudar, o Pod continua rodando normalmente.  
* **nodeSelectorTerms**: lista de condições.  
* **matchExpressions**: define a regra de comparação.  
  * **key**: é o nome da label, nesse caso, node.  
  * **operator**: In indica que o valor da label deve estar dentro da lista definida.
  * **values**: valor esperado da label, nesse caso, cluster1-worker2.

Como a regra de Affinity foi criada para o node com a label **cluster1-worker2**, todos os pods foram implantados nesse node.

<h3 align="center"> 16.7.  Criando regra de AntiAffinity. </h3>

As regras de **AntiAffinity** no Kubernetes servem para evitar que certos Pods fiquem próximos uns dos outros - normalmente no mesmo node ou na mesma zona.

Inicie o deployment **16_pod-antiaffinity.yaml** com o conteúdo de AntiAffinity abaixo:

* **Aplique essa configuração**: kubectl apply -f 16_pod-antiaffinity.yaml

Abaixo, a explicação de cada parâmetro:

* **affinity**: início da regra de AntiAffinity para o Pod.  
* **podAntiAffinity**: define regras para evitar proximidade entre Pods.  
* **requiredDuringSchedulingIgnoreDuringExecution**: regra obrigatória. Se não cumprir, o Pod não será agendado. Depois que o Pod estiver rodando, se a label mudar, o Pod continua rodando normalmente.  
* **labelSelector**: seleciona os Pods que serão evitados.  
* **matchLabels**: faz o "match" com os pods.  
* **app**: os pods com essa label são proibidos de ficar perto.  
* **topologyKey**: define onde a regra se aplica, nesse caso, vai usar o label **node**.

<h3 align="center"> 16.8. Usando o preferredScheduling e o requiredScheduling. </h3>

Os termos **preferredScheduling** e **requiredScheduling** definem como o scheduler trata as regras de Affinity e AntiAffinity.  

* **preferredScheduling**: é uma regra de preferência. 
* **requiredScheduling**: é uma regra obrigatória. Todos os nodes válidos continuam candidatos, mas o scheduler atribui pontuações (scores) e escolhe o node com maior score.

Para teste, copie o arquivo **16_pod-affinity.yaml** para **16_pod-antiaffinity-2.yaml** e altere a configuração para o exemplo abaixo:  
<p align="center"><img width="454" height="149" alt="image" src="https://github.com/user-attachments/assets/346b800e-a4df-475d-8045-130bb866ef04" /></p>

* **Aplique essa configuração**: kubectl apply -f 16_pod-antiaffinity-2.yaml  
 
Abaixo, a explicação de cada parâmetro:

* **affinity**: início da regra de Affinity para o Pod.  
* **podAffinity**: define regras para aproximar Pods.  
* **preferredDuringSchedulingIgnoreDuringExecution**: é uma regra de preferência. O scheduler tenta respeitar, mas pode ignorar.  
* **weight**: peso da preferência. **100** indica prioridade máxima. Nodes que atendem essa regra recebem maior pontuação.  
* **podAffinityTerm**: define a regra de afinidade em si.  
* **labelSelector**: seleciona os pods que queremos ficar próximos.  
* **matchLabels**: labels do Pod para fazer o match.  
  * **app**: seleciona os Pods que queremos ficar próximos.  
* **topologyKey**: define onde a regra se aplica, nesse caso, kubernetes.io/hostname indica que vai usar o mesmo node.  

Usando essa configuração, o scheduler pontuou melhor, o mesmo node, como resultado, todos os pods ficaram juntos no mesmo node.
