<h2 align="center"> Descomplicando Network Policy e o EKS. </h2>

<h3 align="center"> 17.1. Instalando o EKS com o EKSCTL. </h3>

No Kubernetes, uma Network Policy é um conjunto de regras que definem como os Pods podem se comunicar entre si e com outros endpoints de rede. Por padrão, os Pods em um cluster Kubernetes podem se comunicar livremente entre si, o que pode não ser ideal para todos os cenários. As Network Policies permitem que você restrinja esse acesso, garantindo que apenas o tráfego permitido possa fluir entre os Pods ou para/endereços IP externos.

**Para que servem as Network Policies?**  
Networks Policies são usadas para:  
* Isolar Pods de tráfego não autorizado.  
* Controlar o acesso à serviços específicos.  
* Implementar padrões de segurança e conformidade.  

**Conceitos fundamentais: Ingress e Egress.**  
* Ingress: Regras de ingresso controlam o tráfego de entrada para um Pod.  
* Egress: Regras de saída controlam o tráfego de saída de um Pod.  

Ter o entendimento desses conceitos é fundamental para entender como as Network Policies funcionam, pois você precisará especificar se uma regra se aplica ao tráfego de entrada ou de saída.  

**Como funcionam as Network Policies?**  
Network Policies utilizam SELECTORS para identificar grupos de Pods e definir regras de tráfego para eles. A política pode especificar:  

* Ingress (entrada): quais Pods ou endereços IP podem se conectar a Pods selecionados.  
* Egress (saída): para quais Pods ou endereços IP os Pods selecionados podem se conectar.

<h3 align="center"> 17.2. Instalando o Add-on AWS VPC CNI. </h3>

Antes de iniciar a instalação do plugin, verifique em **[Managing-VPC-CNI](https://docs.aws.amazon.com/eks/latest/userguide/managing-vpc-cni.html)** a versão do VPC CNI que devemos instalar em relação a versão do cluster que foi iniciada.

Para iniciar o cluster, execute: 

**eksctl create cluster --name=eks-cluster --version=1.35 --region=us-east-1 --nodegroup-name=eks-cluster-nodegroup --node-type=t3.medium --nodes=2 --nodes-min=1 --nodes-max=3 --managed**  

Como foi instalada a versão do cluster **1.35**, devemos instalar a versão **1.21.1** do VPC CNI.

Para instalar o plugin **AWS VPC CNI**, execute o comando: **eksctl create addon --name vpc-cni --version v1.21.1-eksbuild.1 --cluster eks-cluster --force**  

Após realizar a instalação desses componentes, acesse o painel do EKS na página da AWS, vá em Add-on e verifique se o componente foi instalado.

Após instalado, clique em Editar, vá em **Valores de configuração** e inclua a configuração **"enableNetworkPolicy": "true"**

<h3 align="center"> 17.4.  Criando a nossa primeira Network Policy. </h3>

Segundo a documentação do **[Network Policy](https://kubernetes.io/docs/concepts/services-networking/network-policies/)**, se você deseja controlar o fluxo de tráfego no nível de endereço IP ou porta (camada 3 ou 4 do modelo OSI), as NetworkPolicies permitem especificar regras para o fluxo de tráfego dentro do seu cluster, bem como entre os Pods e o mundo externo. Seu cluster deve usar um plugin de rede que suporte a aplicação de NetworkPolicies.

Realize os passos abaixo para testar a comunicação entre pods.

* **Inicie um Pod**: kubectl run -ti alpine --image alpine -- sh  
* **Adicione o curl**: apk add curl redis
* * **Faça um curl para o serviço**: curl giropops-senhas.giropops.svc:5000  

Se o comando anterior retornar **PONG**, a comunicação ocorreu com sucesso. Após esses comandos, a página será retornada no formato html... isso demonstra que esse Pod conseguiu alcançar o outro, mesmo em namespaces diferentes.  

**Iniciando a primeira Policy.**  
Crie o arquivo **17_network-policy-1.yaml** para criar a primeira Network Policy. Essa política será criada para bloquear o acesso a pods em uma determinada namespace.
<p align="center"><img width="271" height="197" alt="image" src="https://github.com/user-attachments/assets/2d247f77-15ea-4c6e-97d1-6a42f32550b1" /></p>

Com essa configuração, pods fora do namespace giropops não irão conseguir se comunicar com pods nessa namespace.  

* **Aplique essa configuração**: kubectl apply -f 17_network-policy-1.yaml  

Abaixo, a explicação de cada componente dessa policy:

* **apiVersion**: define a versão da API do Kubernetes.  
* **kind**: define o tipo de recurso criado.  
* **metadata**: metadados do objeto.  
  * **name**: nome da NetworkPolicy  
  * **namespace**: define onde a policy será aplicada.  
* **spec**: especificações da política.  
  * **podSelector**: define quais pods serão afetados pela política.  
  * **matchLabels**: seleciona os pods que serão afetados, nesse caso, os pods com a label **app: redis**.  
* **ingress**: define regras de tráfego de entrada.  
  * **from**: define quem pode acessar os pods.  
  * **podSelector**: usar **{}** indica que todos os pods desse namespace (**giropops**) podem acessar o Pod especifico.  

⚠️ O comportamento real essa política indica que ela é aplicada a todos os pods com a label app: redis e permite tráfego de entrada de qualquer pod no namespace giropops... pods em outros namespaces terão o tráfego bloqueado.

Para visualizar as políticas criadas, execute **kubectl get netpol**  
<p align="center"><img width="374" height="35" alt="image" src="https://github.com/user-attachments/assets/66f57e10-6124-44bd-b9c9-7e26f4ca52d5" /></p>

Após essa configuração, faça os testes novamente:

* **Inicie um Pod**: kubectl exec alpine -ti -- sh  
* **Adicione o curl**: apk add curl redis  
* **Faça um curl para o serviço**: curl giropops-senhas.giropops.svc:5000  
* **Faça um ping para o Redis**: redis-cli -h redis-service.giropops.svc ping  

O ping para o redis não vai retornar nada por causa da política criada, porém, o **curl** para a aplicação irá funcionar corretamente pois a política foi criada apenas para o redis.
