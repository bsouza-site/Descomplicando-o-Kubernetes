# Descomplicando os Deployments e estratégias de rollout.
### 03.1. O que é um Deployment?  
O **Deployment** no Kubernetes é um recurso que define o estado desejado de uma aplicação em execução. Ele permite que você descreva, atualize e gerencie aplicações de forma declarativa. Ao utilizar Deployments, você não precisa se preocupar com os detalhes da implementação, pois o Kubernetes cuidará automaticamente do processo de implantação e atualização.

Um Deployment permite que você:

* Defina quantas réplicas da aplicação deve existir;
* Garanta alta disponibilidade;
* Realize atualizações sem downtime (rolling update);
* Faça rollback para versões anteriores;
* Mantenha os pods sempre no estado desejado.

Em resumo: você descreve **o estado desejado**, e o Kubernetes garante que ele seja mantido.  

### 03.2. Criando um Deployment através de um manifesto.
Criar um Deployment via manifesto no Kubernetes significa definir a aplicação em um arquivo YAML e pedir para o Kubernetes criar e gerenciar esse Deployment com base nessa definição.

Para iniciar a criação de um arquivo de manifesto, crie o arquivo **03_deployment.yaml** com o conteúdo abaixo:

* **Para aplicar este deployment, execute**: kubecl apply -f deployment.yaml
* **Para ver o deployment criado**: kubectl get deployments -o wide
* **Para ver mais detalhes desse deployment**: kubectl describe deployments
* **Ver os pods desse deployment pela label**: k get pods -l app=nginx-deployment
* **Para remover um deployment, execute**: kubectl delete -f deployment.yaml

Nas linhas abaixo, vamos entender o que significa cada parâmetro:

* **apiVersion**: indica a versão da API do Kubernetes usada para criar o recurso. A versão v1 é a API estável para criar recursos básicos (Pod, Service, ConfigMap).  
* **kind**: define o tipo de recurso que será criado, nesse caso, **Deployment**.  
* **metadata**: informações relacionadas ao **Deployment**.  
  * **labels**: rótulos usados para organização e seleção.  
  * **name**: nome do Deployment. Deve ser único dentro do namespace.  
* **spec**: especificações de como o Deployment deve se comportar.  
* **replicas**: quantidade de réplicas dos Pods.  
* **selector**: define quais pods pertencem a esse deployment.  
  * **matchLabels**: este campo deve combinar exatamente com os labels dos Pods. Isso evita que o deployment gerencie pods errados.  
* **strategy**: define a estratégia de atualização (rolling update, recreate etc.). Como está vazio, o Kubernetes usa o padrão rolling update, isso atualiza os pods aos poucos, sem downtime.  
* **template**: define o modelo dos Pods que o Deployment cria.  
  * **metadata**: informações relacionadas aos Pods que serão criados.  
  * **labels**: os labels que serão aplicados aos Pods. Precisam ser os mesmos do matchLabels.  
* **spec**: essa segunda especificação, são as especificações dos containeres.

Após essa implantação, os pods foram criados corretamente.

É possível visualizar a saída do comando **kubectl get deployment** no formato YAML com a opção **-o yaml**  

Abaixo, outras opções para trabalhar com deployments:  
* **Para criar um deployment via linha de comando, execute**: kubectl create deployment --image **nginx** --replicas **3 nginx-deployment**  
* **Também é possível usar o dry-run para simular a criação de um deployment**: kubectl create deployment --image **nginx** --replicas **3 nginx-deployment** --dry-run=client  
* **Para gerar um arquivo de deployment, use o dry-run e direcione a saída do comando para um arquivo**: kubectl create deployment --image **nginx** --replicas **3 nginx-deployment** --dry-run=client -o yaml > **new-dep.yaml**
