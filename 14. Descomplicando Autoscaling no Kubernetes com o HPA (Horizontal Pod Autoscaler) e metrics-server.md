<h2 align="center"> Descomplicando Autoscaling no Kubernetes com o HPA (Horizontal Pod Autoscaler) e metrics-server. </h2>

<h3 align="center"> 14.1. O que é Horizontal Pod Autoscaler. </h3>

O HPA (Horizontal Pod Autoscaler) no Kubernetes é um recurso que aumenta ou diminui automaticamente a quantidade de Pods de uma aplicação, com base em métricas de uso.

Com o HPA, podemos ajustar automaticamente o número de réplicas de um conjunto de pods, assegurando que nosso aplicativo tenha sempre os recursos necessários para performar eficientemente, sem desperdiçar recursos. O HPA é como um maestro que, com a batuta das métricas, rege a orquestra de pods, assegurando que a harmonia seja mantida mesmo quando a sinfonia do tráfego de rede atinge seu crescendo.

E onde o **metrics-server** se encaixa nisso? O metrics-server é responsável por fornecer métricas básicas (CPU e memória) para o HPA.

<h3 align="center"> 14.2. Instalando o metrics-server. </h3>

**Introdução.**  
O **metrics-server** é um componente do Kubernetes responsável por coletar e disponibilizar métricas básicas de uso dos nós e pods dentro do cluster.

O metrics-server é uma fonte escalável e eficiente de métricas de recursos de contêineres para pipelines de escalonamento automático integrados ao Kubernetes.

**Instalação.**  
A versão mais recente do metrics-server pode ser instalada executando o seguinte comando:

* kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

<p align="center"><img width="274" height="290" alt="image" src="https://github.com/user-attachments/assets/a1d3ec17-53cd-4c6c-8420-0c2410bd8ed3" /></p>

Para entender as configurações utilizadas:

* **apiVersion**: define a versão da API do HPA.  
* **kind**: define o tipo de recurso, nesse caso, **HorizontalPodAutoscaler**  
* **metadata**: contém informações básicas do objeto, como **name nginx-hpa**  
* **spec**: especificações do objeto.  
  * **scaleTargetRef**: define qual recurso será escalado.  
  * **apiVersion: apps/v1**: versão da API do recurso alvo (Deployment).  
  * **kind**: tipo de recurso que será escalado.  
  * **name**: nome do Deployment que será escalado.  
* **minReplicas**: número mínimo de Pods que sempre estarão rodando.  
* **maxReplicas**: número máximo de pods que podem ser criados.  
* **metrics**: define com base em qual métrica o Autoscaling vai acontecer.  
  * **type**: tipo de métrica. **Resource** significa que estamos usando recursos do próprio container (CPU ou memória).  
  * **name**: define que a métrica usada é CPU.  
* **target**: define a meta que o HPA tenta manter.  
  * **type**: tipo de meta. **Utilization** indica o percentual de uso baseado no request de CPU.  
  * **averageUtilization**: meta de uso médio. O HPA tenta manter a média em 50%.  

Após realizar essa configuração:

* **Para aplicar essa configuração, execute**: kubectl apply -f 14_hpa-1.yaml  
* **Para ver os HPAs existentes**: kubectl get hpa

<h3 align="center"> 14.4. Instalando o Locust para o nosso Stress Test. </h3>

**O que é Locust?**  
O Locust é uma ferramenta de teste de carga (load testing) usada para simular usuários acessando uma aplicação e verificar como ela se comporta sob pressão.  

⚙️ O Locust simula vários usuários virtuais que executam tarefas como:  

* Acessar páginas HTTP;  
* Fazer login;  
* Enviar requisições para APIs.  

Esses comportamentos são escritos em Python, o que dá muita flexibilidade.  

Antes de iniciar a implantação do Pod do Locust, realize as configurações abaixo:

<h3 align="center"> 14.6. Utilizando configurações avançadas para o ScaleUp e ScaleDown usando HPA. </h3>

Volte ao arquivo **14_hpa-1.yaml** e realize as configurações abaixo:
<p align="center"><img width="304" height="598" alt="image" src="https://github.com/user-attachments/assets/522bf0e1-6d99-4377-9629-72c2be28d3fd" /></p>

Para entender as configurações acima:

* **behavior**: define as regras de escalonamento.  
* **scaleUP**: controla como o número de pods aumenta.  
  * **stabilizationWindowSeconds**: janela de estabilização de **5 segundos**. O HPA olha os últimos **5s** antes de decidir escalar.  
  * **policies**: define limites de crescimento.  
  * **type**: o percentual será baseado em **percentual** (Percent).  
  * **value**: pode aumentar até **100%** do número atual de pods.  
  * **periodSeconds**: esse limite vale para cada janela de 10 segundos.  
* **ScaleDown**: controla como o número de pods diminui.  
  * **stabilizationWindowSeconds**: janela de estabilização de **10 segundos**. Neste caso, é o tempo que o HPA espera para começar a reduzir Pods.  
  * **policies**: define limites, nesse caso, para redução.  
  * **type**: o percentual será baseado em **percentual** (Percent).  
  * **value**: pode reduzir até **100%** dos pods. Por exemplo, se tem 10 pods, o número pode cair direto para **3**, que é o número mínimo configurado.  
  * **periodSeconds**: pode aplicar essa redução a cada **10** segundos.
