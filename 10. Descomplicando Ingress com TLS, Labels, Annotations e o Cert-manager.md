<h3 align="center"> 10.1. O que é o Cert-Manager?</h3>
O cert-manager cria certificados TLS para cargas de trabalho em seu cluster Kubernetes ou OpenShift e renova os certificados antes que expirem.

O cert-manager pode obter certificados de diversas autoridades certificadoras, incluindo: Let's Encrypt, HashiCorp Vault, CyberArk Certificate Manager e PKI privada.

Com o recurso Certificate do cert-manager, a chave privada e o certificado são armazenados em um Kubernetes Secret, que é montado por um Pod da aplicação ou usado por um controlador Ingress. Com o csi-driver, csi-driver-spiffe ou istio-csr, a chave privada é gerada sob demanda, antes da inicialização da aplicação; a chave privada nunca sai do nó e não é armazenada em um Kubernetes Secret.

**Issuer e ClusterIssuer.**  
No cert-manager, **Issuer** e **ClusterIssuer** são recursos do Kubernetes que representam autoridades de certificação (CAs) capazes de gerar certificados assinados ao atenderem solicitações de assinatura de certificados. Todos os certificados do cert-manager exigem uma autoridade emissora referenciada que esteja pronta para tentar atender à solicitação.

**Issuer.**  
Um certificado Issuer é um recurso com namespace, e não é possível emitir certificados a partir de um certificado Issuer em um namespace diferente. Isso significa que você precisará criar um certificado Issuer em cada namespace no qual deseja obter certificados.

**ClusterIssuer.**  
Se você deseja criar uma única instância Issuer que possa ser consumida em vários namespaces, considere criar um recurso ClusterIssuer. Ele é quase idêntico ao recurso Issuer padrão, porém não está vinculado a um namespace, podendo ser usado para emitir solicitações de Certificates em todos os namespaces.

Abaixo, o entendimento de cada campo da configuração desses arquivos:

* **apiVersion**: define a versão da API do Cert-Manager que está sendo utilizada.  
* **kind**: define o tipo de recurso:  
  * **Issuer**: emissor válido apenas dentro de um namespace.  
  * **ClusterIssuer**: válido para o cluster inteiro.  
* **metadata**: metadados do recurso Kubernetes.  
  * **name**: nome do Issuer.  
* **spec**: especificação do emissor.  
  * **server**: define a URL do servidor ACME. Note que é diferente para Issuer e ClusterIssuer.  
  * **e-mail**: seu e-mail.  
* **privateKeySecretRef**: define o Secret onde será armazenada a chave privada da conta ACME.  
* **solvers**: define como o cert-manager provará que você é dono do domínio.  
  * **http01**: define que será usado o HTTP-01 challenge.  
  * **ingress**: Define qual controller de Ingress será usado para resolver o challenge.  
  * **ingressClassName**: vai usar o IngressClassName nginx.  

Aplique essa configuração com os comandos:

* kubectl apply -f 10_staging_issuer.yaml
* kubectl apply -f 10_production_issuer.yaml

Verifique se as configurações de Issuer e ClusterIssuer foram aplicadas corretamente:

* kubectl get issuers.cert-manager.io

Abaixo, alguns comandos usados nessa aula.  
**kubectl get certificate**  
O comando **kubectl get certificate** lista os certificados criados pelo Cert-Manager.  

**kubectl get certificaterequest**  
O comando **kubectl get certificaterequest** lista os CertificateRequests, que é o pedido interno que o cert-manager cria automaticamente quando você cria um Certificate.  

**kubectl get secret**  
O comando **kubectl get secret** lista todos os secrets criados em um namespace.  

**kubectl get orders.acme.cert-manager.io**  
O comando **kubectl get orders.acme.cert-manager.io** lista os Orders ACME existentes. Esse recurso representa a ordem enviada para a ACME. Se o Order (pedido) falhar, o certificado não será emitido.
