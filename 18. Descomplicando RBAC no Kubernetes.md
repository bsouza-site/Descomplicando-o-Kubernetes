<h2 align="center"> Descomplicando RBAC no Kubernetes. </h2>

<h3 align="center"> 18.1.	O que é RBAC, Role, RoleBinding, ClusterRole e ClusterRoleBinding. </h3>

**RBAC.**    
RBAC é um acrônimo para Role-Based Access Control, ou Controle de Acesso Baseado em Funções. É um método de controle de acesso que permite que um administrador defina permissões específicas para usuários e grupos de usuários. Isso significa que os administradores podem controlar quem tem acesso a quais recursos e o que eles podem fazer com esses recursos.  

No Kubernetes é super importante você entender como funciona o RBAC, pois é através dele que você vai definir as permissões de acesso aos recursos do cluster, como por exemplo, quem pode criar um Pod, um Deployment, um Service, etc.  

**Role.**  
Uma Role define permissões dentro de um namespace específico, por exemplo, permitir ler pods apenas em um namespace específico. Uma Role sempre define permissões dentro de um namespace; ao criar uma Role, você precisa especificar o namespace ao qual ela pertence.  

**RoleBinding.**  
Um RoleBinding conecta uma Role a um usuário, grupo ou ServiceAccount. Uma RoleBinding concede as permissões definidas em uma Role a um usuário ou conjunto de usuários.  

**ClusterRole.**  
Um ClusterRole define permissões no cluster inteiro (ou reutilizável em vários namespaces). Os recursos têm nomes diferentes (Role e ClusterRole) porque um objeto do Kubernetes sempre precisa ser ou ter um namespace ou não; não pode ser ambos.  

**ClusterRoleBinding.**  
Um ClusterRoleBinding conecta um ClusterRole a um usuário em todo o cluster.  

<h3 align="center"> 18.2. Criando e assinando um certificado no Kubernetes. </h3>

Nessa aula, será criado um certificado de autenticação de cliente. Uma recomendação é, se for criado um novo usuário, usar certificados, se for criado um service account, criar um Token.  

* **Para criar uma nova key, execute**: openssl genrsa -out developer.key 2048  
* **Crie a key para CSR**: openssl req -new -key developer.key -out developer.csr -subj "/CN=developer"  
* **Converter o CSR para base64**: cat developer.csr | base64 | tr -d '\n'  

Lembrando que **developer** nos passos acima é o “usuário” e pode ser personalizado. Copie a saída desse último comando pois ele será usado no arquivo CSR no Kubernetes.

* **Aplique essa configuração**: kubectl apply -f 18_developer.yaml  

Abaixo, a explicação dos parâmetros usados:  

* **apiVersion**: define a API usada para certificados no Kubernetes.  
* **kind**: tipo de recurso, nesse caso, **CertificateSigningRequest**.  
* **metadata**: dados informados na requisição.  
  * **name**: nesse caso, o nome do CSR dentro do Cluster.  
* **spec**: especificações do CSR.  
  * **request**: a chave do CSR que foi codificado em base64.  
* **signerName**: define que vai assinar o certificado. Este vai ser um certificado para autenticação de cliente na API do Kubernetes.  
* **expirationSeconds**: tempo de validade do certificado, nesse caso, 1 ano em segundos.  
* **usages**: define para que o certificado será usado, nesse caso, autenticação de cliente.  

**Para ver os CSRs criados**: kubectl get csr  

**kubectl certificate**  
Neste momento, devemos aprovar a solicitação de certificado. O comando **kubectl certificate** no Kubernetes serve para gerenciar pedidos de certificados digitais (CSRs) dentro do Kubernetes.  

Utilizando o comando **kubectl certificate**, podem ser usados duas opções:  

* **approve**: Aprovar uma solicitação de assinatura de certificado.  
* **deny**: negar uma solicitação de assinatura de certificado.  

Para esse caso, aprove a assinatura com o comando **kubectl certificate approve developer**

**Agora que o certificado foi aprovado, devemos extrair o certificado propriamente... para isso, execute**: kubectl get csr developer -o jsonpath='{.status.certificate}' | base64 --decode > developer.crt
