<h2 align="center"> Descomplicando RBAC no Kubernetes. </h2>

<h3 align="center"> 18.1.	O que é RBAC, Role, RoleBinding, ClusterRole e ClusterRoleBinding. </h3>

**RBAC.**    
RBAC é um acrônimo para Role-Based Access Control, ou Controle de Acesso Baseado em Funções. É um método de controle de acesso que permite que um administrador defina permissões específicas para usuários e grupos de usuários. Isso significa que os administradores podem controlar quem tem acesso a quais recursos e o que eles podem fazer com esses recursos.  

No Kubernetes é super importante você entender como funciona o RBAC, pois é através dele que você vai definir as permissões de acesso aos recursos do cluster, como por exemplo, quem pode criar um Pod, um Deployment, um Service, etc.  

**Role.**  
Uma Role define permissões dentro de um namespace específico, por exemplo, permitir ler pods apenas em um namespace específico. Uma Role sempre define permissões dentro de um namespace; ao criar uma Role, você precisa especificar o namespace ao qual ela pertence.  

**RoleBinding.**  
Um RoleBinding conecta uma Role a um usuário, grupo ou ServiceAccount. Uma RoleBinding concede as permissões definidas em uma Role a um usuário ou conjunto de usuários.  

**ClusterRole.**  
Um ClusterRole define permissões no cluster inteiro (ou reutilizável em vários namespaces). Os recursos têm nomes diferentes (Role e ClusterRole) porque um objeto do Kubernetes sempre precisa ser ou ter um namespace ou não; não pode ser ambos.  

**ClusterRoleBinding.**  
Um ClusterRoleBinding conecta um ClusterRole a um usuário em todo o cluster.  

<h3 align="center"> 18.2. Criando e assinando um certificado no Kubernetes. </h3>

Nessa aula, será criado um certificado de autenticação de cliente. Uma recomendação é, se for criado um novo usuário, usar certificados, se for criado um service account, criar um Token.  

* **Para criar uma nova key, execute**: openssl genrsa -out developer.key 2048  
* **Crie a key para CSR**: openssl req -new -key developer.key -out developer.csr -subj "/CN=developer"  
* **Converter o CSR para base64**: cat developer.csr | base64 | tr -d '\n'  

Lembrando que **developer** nos passos acima é o “usuário” e pode ser personalizado. Copie a saída desse último comando pois ele será usado no arquivo CSR no Kubernetes.  

**Criar um CSR no Kubernetes.**  
Agora, é necessário criar o Certificate Signing Request (CSR), que significa pedido de certificado digital no Kubernetes... para isso, crie o arquivo **18_developer.yaml** com o conteúdo abaixo:  
<p align="center"><img width="413" height="262" alt="image" src="https://github.com/user-attachments/assets/94a7e19a-6ea8-46cf-8fe7-b4607f120fb7" /></p>

* **Aplique essa configuração**: kubectl apply -f 18_developer.yaml  

Abaixo, a explicação dos parâmetros usados:  

* **apiVersion**: define a API usada para certificados no Kubernetes.  
* **kind**: tipo de recurso, nesse caso, **CertificateSigningRequest**.  
* **metadata**: dados informados na requisição.  
  * **name**: nesse caso, o nome do CSR dentro do Cluster.  
* **spec**: especificações do CSR.  
  * **request**: a chave do CSR que foi codificado em base64.  
* **signerName**: define que vai assinar o certificado. Este vai ser um certificado para autenticação de cliente na API do Kubernetes.  
* **expirationSeconds**: tempo de validade do certificado, nesse caso, 1 ano em segundos.  
* **usages**: define para que o certificado será usado, nesse caso, autenticação de cliente.  

**Para ver os CSRs criados**: kubectl get csr  

**kubectl certificate**  
Neste momento, devemos aprovar a solicitação de certificado. O comando **kubectl certificate** no Kubernetes serve para gerenciar pedidos de certificados digitais (CSRs) dentro do Kubernetes.  

Utilizando o comando **kubectl certificate**, podem ser usados duas opções:  

* **approve**: Aprovar uma solicitação de assinatura de certificado.  
* **deny**: negar uma solicitação de assinatura de certificado.  

Para esse caso, aprove a assinatura com o comando **kubectl certificate approve developer**

**Agora que o certificado foi aprovado, devemos extrair o certificado propriamente... para isso, execute**: kubectl get csr developer -o jsonpath='{.status.certificate}' | base64 --decode > developer.crt  

<h3 align="center"> 18.3. Api-resources e criando a nossa primeira Role. </h3>

**kubectl api-resources.**  
O comando **kubectl api-resources** é usado no Kubernetes para listar todos os recursos da API disponíveis no cluster, ou seja, tudo aquilo que você pode criar, consultar ou gerenciar com o kubectl.  
<p align="center"><img width="944" height="145" alt="image" src="https://github.com/user-attachments/assets/cadb9c74-49ab-4c01-a734-42c18169f6c5" /></p>

Esse comando é extremamente importante para descobrir os recursos disponíveis, ver os alias que podem ser usados, entre outras informações... abaixo, o significado de cada campo:  
* **NAME**: nome do recurso.  
* **SHORTNAMES**: alias que pode ser usados.  
* **APIVERSION**: versão da API. Este campo é interessante, pois indica a versão da API atual... essa informação pode ser usada ao criar manifestos.  
* **NAMESPACED**: se o recurso pertence a um namespace. Em casos de **false**, indica que o recurso funciona para o cluster todo.  
* **KIND**: tipo de objeto na API.  

Abaixo, alguns parâmetros que podem ser usados com esse comando:  
* **Filtrar por namespaced true ou false**: kubectl api-resources --namespaced  
* **Mostrar mais detalhes**: kubectl api-resources **-o wide**  

Utilizando o parâmetro **-o wide**, é apresentado o campo **VERBS**. Esse campo mostra a lista de ações suportadas pela API para aquele recurso, como no caso de um Pod, é possível executar as ações **create,delete,deletecollection,get,list,patch,update,watch**

<h3 align="center"> 18.5. Configurando o nosso User e um novo contexto com o kubectl config. </h3>

**kubectl config**  
O comando **kubectl config** (já visto em outras partes do curso) é usado para gerenciar a configuração de acesso ao cluster Kubernetes, ou seja, tudo relacionado a clusters, usuários e contextos definidos no arquivo kubeconfig.  

📌 **O que é o kubeconfig?**  
É um arquivo (geralmente em ~/.kube/config) que armazena:  

* Clusters (endereços dos clusters Kubernetes);  
* Usuários (credenciais/certificados/tokens);  
* Contexts (combinação de cluster + usuário + namespace).  

**Configurar as credenciais.**  
Para informar as credenciais do usuário **developer** criado anteriormente, utilize o comando:   

**kubectl config set-credentials developer --client-certificate developer.crt –client-key developer.key --embed-certs**  

* **Para ver os usuários criados, execute** kubectl config get-users  
* **Ver os usuários de um cluster especifico**: kubectl config get-users --context **cluster1**  
* **Agora, configure o contexto para o usuário developer**: kubectl config set-context **developer-aula** --cluster **cluster1** --namespace **dev** --user **developer**  
* **Para trabalhar com o novo contexto**: kubectl config use-context **developer-aula**

Após isso, é possível realizar apenas as ações configuradas no arquivo de role, como criar e listar Pods no namespace indicado no arquivo (**dev**).

**Alterar permissões.**  
A role criada anteriormente não permite coisas como deletar pods (mesmo os Pods criados por mim mesmo) ou visualizar informações de deployments.

Foi modificado a role criada nessa aula para que seja possível listar, criar, visualizar e remover pods e deployments.

<h3 align="center"> 18.7.	Criando um ServiceAccount utilizando Token. </h3>

Uma das formas de autenticação no Kubernetes é através de Tokens, que são utilizados para autenticar Service Accounts, que são contas de serviço utilizadas, normalmente, por aplicações que rodam no cluster ou serviços que precisam acessar o cluster.

Os Tokens são gerados automaticamente pelo Kubernetes, e são utilizados para autenticar Service Accounts, e eles são armazenados em Secrets, que são recursos do Kubernetes que armazenam informações sensíveis, como por exemplo, chaves privadas, senhas, tokens, etc.
